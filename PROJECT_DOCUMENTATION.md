PROJECT OVERVIEW
================================================================================
The AI Requirement Engineering Assistant is a modern web application that uses
artificial intelligence to automatically generate comprehensive requirement 
artifacts from raw text or uploaded documents. Users can extract requirements,
generate user stories, create acceptance criteria, produce test cases, evaluate
quality, build traceability matrices, and export all artifacts in JSON format.

Originally built with Streamlit, the project has been modernized to use a
React frontend with a FastAPI backend, providing a superior user experience
and scalable architecture.

TECHNOLOGY STACK
================================================================================

BACKEND:
- Framework: FastAPI (Python 3.x web framework)
- Server: Uvicorn (ASGI server)
- LLM Integration: Groq and Gemini APIs with fallback implementations
- Authentication: JWT tokens with 24-hour expiry
- Data Storage: JSON-based user credentials (data/users.json)
- Password Security: SHA256 hashing
- API Documentation: Swagger UI (auto-generated by FastAPI)
- CORS: Enabled for localhost:5173 (React development server)

FRONTEND:
- Framework: React 18.2.0
- Routing: React Router 6.12.1
- HTTP Client: Axios
- Build Tool: Vite 5.0.0 (dev server on port 5173)
- Styling: CSS with dark gradient theme
- State Management: React Hooks (useState, useEffect)

RUNTIME:
- Backend: http://127.0.0.1:8000
- Frontend: http://localhost:5173

CORE FEATURES & ENDPOINTS
================================================================================

1. AUTHENTICATION
   ├── POST /api/auth/login
   │   Input: {username, password}
   │   Output: {access_token, token_type}
   │   Status: 200 Success / 401 Unauthorized
   │
   └── POST /api/auth/register
       Input: {username, password, email (optional)}
       Output: {status, message}
       Status: 200 Success / 400 User exists

2. DOCUMENT UPLOAD
   ├── POST /api/upload
   │   Input: File (multipart/form-data)
   │   Output: {filename, size}
   │   Status: 200 Success

   └── POST /api/llm/query
       Input: {prompt, context}
       Output: {response}

3. ARTIFACT GENERATION (7 Endpoints in /api/features/)

   a) Requirements Extraction
      POST /api/requirements/extract
      Input: {text}
      Output: {requirements: [{id, title, description, type, priority}]}

   b) User Stories Generation
      POST /api/features/user-stories
      Input: {text}
      Output: {requirements, user_stories: [{id, requirement_id, as_a, i_want, so_that}]}

   c) Acceptance Criteria
      POST /api/features/acceptance-criteria
      Input: {text}
      Output: {requirements, user_stories, acceptance_criteria: [{id, story_id, criteria}]}

   d) Test Cases
      POST /api/features/test-cases
      Input: {text}
      Output: {requirements, testcases: [{id, requirement_id, description, steps, expected_result}]}

   e) Quality Evaluation
      POST /api/features/quality
      Input: {text}
      Output: {requirements, quality_findings: [{id, issue_type, detail, recommendation}]}

   f) Traceability Matrix
      POST /api/features/traceability
      Input: {text}
      Output: {requirements, user_stories, testcases, traceability_matrix: [{requirement_id, user_story_id, testcase_id}]}

   g) Full Workflow (Orchestrated Pipeline)
      POST /api/features/full-workflow
      Input: {text}
      Output: {requirements, user_stories, acceptance_criteria, testcases, quality_findings, traceability_matrix}

4. EXPORT
   ├── POST /api/features/export
   │   Input: {all artifacts}
   │   Output: {JSON-formatted artifacts}
   │
   └── Frontend Download
       Formats: JSON (fully implemented), Word/PDF (stubs)

FEATURE EXPLANATION
================================================================================

This section explains what each major feature does in the app and what output
the user receives after using it.

1. Overview
   What it does:
   ├── Shows the dashboard landing area.
   ├── Displays the backend health check.
   └── Introduces the available workflow actions.

   Output:
   ├── API health status message.
   └── Quick start and available actions summary.

2. Upload
   What it does:
   ├── Lets the user select a document file.
   ├── Uploads the file to the backend.
   ├── Parses the uploaded document into text.
   └── Reveals the Generate Analysis button after upload.

   Output:
   ├── Uploaded file name.
   ├── File size.
   ├── Parsed document text.
   └── Upload success or error message.

3. Generate Analysis from Upload
   What it does:
   ├── Uses the parsed document text from the uploaded file.
   ├── Runs the full requirement engineering workflow.
   └── Shows a loading spinner while the analysis is running.

   Output:
   ├── Requirements.
   ├── User stories.
   ├── Acceptance criteria.
   ├── Test cases.
   ├── Quality findings.
   └── Traceability links.

4. Full Workflow
   What it does:
   ├── Runs the complete pipeline from one input text.
   ├── Generates every artifact in one call.
   └── Gives the fastest end-to-end result for a requirement document.

   Output:
   ├── requirements[]
   ├── user_stories[]
   ├── acceptance_criteria[]
   ├── testcases[]
   ├── quality_findings[]
   └── traceability_matrix[]

5. Requirements
   What it does:
   ├── Breaks raw text into structured requirement items.
   ├── Assigns requirement IDs.
   └── Organizes each requirement with a title and priority.

   Output:
   ├── Requirement ID
   ├── Title
   ├── Description
   ├── Type
   └── Priority

6. User Stories
   What it does:
   ├── Converts requirements into user stories.
   ├── Follows the standard As a / I want / So that format.
   └── Links each story back to its source requirement.

   Output:
   ├── Story ID
   ├── Requirement ID
   ├── As a
   ├── I want
   └── So that

7. Acceptance Criteria
   What it does:
   ├── Turns user stories into acceptance criteria.
   ├── Defines what must be true for the story to be complete.
   └── Makes the story testable and verifiable.

   Output:
   ├── Acceptance ID
   ├── Story ID
   └── Criteria text

8. Test Cases
   What it does:
   ├── Creates test cases from requirements.
   ├── Suggests validation steps.
   └── Shows the expected result for each case.

   Output:
   ├── Test case ID
   ├── Requirement ID
   ├── Description
   ├── Steps
   └── Expected result

9. Quality Report
   What it does:
   ├── Reviews requirements for gaps and ambiguity.
   ├── Flags unclear or incomplete statements.
   └── Gives improvement recommendations.

   Output:
   ├── Finding ID
   ├── Issue type
   ├── Detail
   └── Recommendation

10. Traceability
    What it does:
    ├── Connects requirements to user stories and test cases.
    ├── Shows coverage across the artifact chain.
    └── Helps with auditability and completeness checks.

    Output:
    ├── Requirement ID
    ├── User story ID
    └── Test case ID

11. Export
    What it does:
    ├── Collects all generated artifacts.
    ├── Prepares the output for sharing or archiving.
    └── Supports JSON download from the UI.

    Output:
    ├── JSON export file.
    └── Ready for future Word, PDF, or Excel export support.

AUTHENTICATION FLOW
================================================================================

1. USER REGISTRATION
   ├── Fill: Username, Password, Email (optional)
   ├── POST /api/auth/register
   ├── Backend: Hash password (SHA256)
   ├── Backend: Save to data/users.json
   └── Response: Success message → Redirect to login

2. USER LOGIN
   ├── Fill: Username, Password
   ├── POST /api/auth/login
   ├── Backend: Verify password hash
   ├── Backend: Generate JWT token (24-hour expiry)
   ├── Frontend: Store token in localStorage['access_token']
   ├── Frontend: Navigate to /dashboard
   └── Response: {access_token, token_type: 'bearer'}

3. PROTECTED ROUTE
   ├── ProtectedRoute component checks localStorage
   ├── If no token: Redirect to login (/
   ├── If token exists: Load Dashboard
   └── Logout: Clear token from localStorage

SERVICES LAYER ARCHITECTURE
================================================================================

LLMService (services/llm_service.py)
├── Purpose: Unified LLM interface with fallbacks
├── Providers:
│   ├── Groq API (Primary - Fast, cheaper)
│   ├── Gemini API (Secondary fallback)
│   └── Hardcoded Response (Tertiary fallback - app works without API keys)
└── Methods: query(prompt, context)

RequirementService (services/requirement_service.py)
├── Purpose: Extract requirements from raw text
├── Input: Raw text/document content
├── Process:
│   ├── Try: Call LLM to parse and structure
│   ├── Fallback: Parse text as bullet points
│   └── Generate: REQ-1, REQ-2, ... with titles, descriptions, types
├── Output: [{id, title, description, type, priority}]
└── Fallback: Generic requirement if LLM fails

UserStoryService (services/user_story_service.py)
├── Purpose: Generate user stories from requirements
├── Input: List of requirements
├── Process:
│   ├── For each requirement: Generate user story
│   ├── Format: "As a <role>, I want <feature>, so that <benefit>"
│   └── Generate: US-1, US-2, ...
├── Output: [{id, requirement_id, as_a, i_want, so_that}]
└── Fallback: Template stories if LLM fails

AcceptanceService (services/acceptance_service.py)
├── Purpose: Create acceptance criteria from user stories
├── Input: List of user stories
├── Process:
│   ├── For each user story: Define acceptance conditions
│   ├── Generate: AC-1, AC-2, ...
│   └── Format: Given/When/Then (BDD format)
├── Output: [{id, story_id, criteria}]
└── Fallback: Generic criteria if LLM fails

TestcaseService (services/testcase_service.py)
├── Purpose: Generate test cases from requirements
├── Input: List of requirements
├── Process:
│   ├── For each requirement: Create test scenario
│   ├── Generate: TC-1, TC-2, ...
│   └── Include: Description, steps, expected results
├── Output: [{id, requirement_id, description, steps, expected_result}]
└── Fallback: 3-step standard test if LLM fails

QualityService (services/quality_service.py)
├── Purpose: Evaluate quality and identify gaps
├── Input: List of requirements
├── Process:
│   ├── Analyze: Completeness, clarity, feasibility
│   ├── Generate: QC-1, QC-2, ...
│   └── Types: [Gap, Ambiguity, Risk, Recommendation]
├── Output: [{id, issue_type, detail, recommendation}]
└── Fallback: Generic quality finding if LLM fails

TraceabilityService (services/traceability_service.py)
├── Purpose: Link requirements to user stories to test cases
├── Input: requirements, user_stories, testcases
├── Process:
│   ├── Match: Each requirement to corresponding user story
│   ├── Match: Each user story to corresponding test case
│   └── Generate: Traceability links
├── Output: [{requirement_id, user_story_id, testcase_id}]
└── Fallback: Simple 1:1:1 mapping if LLM fails

FALLBACK LOGIC
================================================================================

Why Fallbacks?
├── LLM APIs have rate limits and may fail
├── API keys may not be configured
├── Network issues may occur
└── App should remain functional without external services

Fallback Chain:
1. Try Groq API
   └─ If fail: Try Gemini API
      └─ If fail: Use hardcoded fallback response
         └─ Result: App works for demos/testing

Fallback Response Example:
├── For requirements: "Create user account" parsed as bullet points
├── For user stories: "As a user, I want to manage my account"
├── For test cases: Standard 3-step test (Open app, Execute, Verify)
├── For quality: Generic "Review for edge cases" recommendation
└── All artifacts include auto-generated IDs (REQ-1, US-1, TC-1, etc.)

ERROR HANDLING
================================================================================

Frontend:
├── Network errors: Displayed to user as error messages
├── Auth failures: Redirect to login
├── Validation: Client-side form validation
└── Loading states: Disabled buttons while processing

Backend:
├── 400 Bad Request: Invalid input
├── 401 Unauthorized: Invalid credentials / Missing token
├── 422 Unprocessable Entity: Validation error
